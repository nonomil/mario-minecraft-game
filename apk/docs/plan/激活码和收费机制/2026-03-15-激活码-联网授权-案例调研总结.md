# 私域分发 APK：联网激活码/授权系统「案例调研 + 落地选型」总结（Research）

> 目标：为“小红书/咸鱼售卖 APK（私域分发）”场景，调研业内常见的**联网试用 + 在线激活（License Server）**做法，整理可直接落地的模式、优缺点、以及建议你优先采用的组合方案。  
> 说明：下文包含第三方服务/产品作为“案例参考”，并不强制选用；你也可以完全自建实现同样的流程。

## 1. 你这个场景的关键矛盾（为什么必须联网）

私域分发常见痛点是：

- **APK 被转发**：只要有人拿到安装包，可能多人共享使用。
- **重装无限试用**：离线试用很难彻底防“清数据/卸载重装”。
- **倒卖/共享激活码**：一个码被多人用，售后极其难控。

因此，只要你已经确认“必须联网”，最稳的方向就是：**试用也上云记账 + 激活绑定设备 + 令牌短有效期 + 服务端可撤销**。

## 2. 可落地模式总览（选型地图）

> 下面不是互斥关系；你可以组合使用。

| 模式 | 你要做的事 | 成本/运维 | 好处 | 坑点/风险 | 适配私域售卖 |
|---|---|---:|---|---|---|
| A. 自建 License Server（推荐） | 写一个轻 API + 简单后台/脚本 | 低（可用免费额度） | 完全可控、可封禁撤销、可统计 | 要自己写/维护 | 很适配 |
| B. 第三方 License 平台（Keygen/Cryptlex/LicenseSpring 等） | 用它们的 API/SDK | 中（通常付费） | 功能齐、上线快、少踩坑 | 平台依赖/费用/网络 | 适配，但看预算 |
| C. 电商 + License Keys（Lemon Squeezy 等） | 用电商卖 key + 授权 API | 中 | “卖货-发码-授权”一体 | 不一定适合国内私域支付链路 | 参考价值大 |
| D. 完整性/反篡改信号（Play Integrity 等） | 服务端验签完整性回执 | 低~中 | 增加破解门槛 | 不是授权系统本身 | 可作为加分项 |

结论：你想要“最便宜、好管理、能控转发/倒卖”，通常走 **A（自建）为主**，再视情况加 **D（完整性信号）**。

## 3. 通用落地架构（抽象成 6 个模块）

不管参考谁的产品/案例，落地时建议都拆成 6 个模块：

1. **设备标识 `device_id`**：客户端通过 native 能拿到的稳定标识做哈希（只传 hash，不传隐私原值）。
2. **试用记账 `trial_state`**：试用额度由服务端按 `device_id` 管，杜绝“重装无限试用”。
3. **激活码 `activation_code`**：高熵随机码；服务端只存 `hash(code)`；支持状态（未用/已用/冻结/撤销）。
4. **绑定关系 `license ↔ device`**：1 个 license 绑定 1 台设备（可配换机次数）。
5. **短期令牌 `access_token`**：激活或刷新后下发，带 `exp`；客户端离线只允许在宽限期内使用。
6. **风控与后台**：限流、日志、封禁、重置绑定（售后核心）。

> 关键安全点：**客户端不能持有任何“能造假”的管理密钥**。所有“能生成/撤销授权”的能力都必须在服务端。

## 4. 案例 1：Keygen（API-First 授权平台）可以借鉴什么

### 4.1 这个案例的核心做法

- 以 API 为中心管理 License：典型流程是 **创建 license → 激活到机器（machine activation）→ 校验/续期**。
- “机器”通常由你提供一个 `fingerprint`（设备指纹）来标识并绑定。
- 官方文档明确强调：**不要把管理用 token/密钥放到客户端**，否则用户可直接伪造请求。

### 4.2 借鉴点（即使你不使用 Keygen）

- **API 形态**：把“激活、验证、刷新、撤销、换机”都变成服务端状态机，客户端只消费结果。
- **激活码不做算法推导**：用随机码 + 服务端查表，避免被逆向出“万能码生成器”。
- **把换机当作一等公民**：把 `rebind_quota` 做到模型里，而不是临时人工处理。

### 4.3 适配私域售卖的优缺点

- 优点：少踩坑、上手快、授权/设备绑定/撤销这些能力齐全。
- 缺点：通常是付费平台；同时在国内私域环境要考虑网络可用性与稳定性。

参考：
- Keygen 文档与指南（含“不要在客户端嵌入 token”提醒）：https://keygen.sh/docs/guides/  
- Keygen 机器激活（machine activation）文档：https://keygen.sh/docs/api/machine-activations/  
- Keygen 示例：Node.js activation server（可借鉴 admin/server 分离的思路）：https://github.com/keygen-sh/example-activation-server-node  

## 5. 案例 2：Cryptlex（LexActivator）主打“节点锁定 + 设备转移”

### 5.1 这个案例的核心做法

- 常见授权形态是 **Node-locked license（节点锁定）**：把授权与设备绑定，支持“设备转移/释放”。
- 提供 SDK/接口来做：
  - 激活（绑定到设备）
  - 校验（含到期/吊销状态）
  - 设备转移（换机/解绑）
- 他们也有“试用/验证试用”的相关能力（便于做服务器记账的 trial）。

### 5.2 借鉴点

- **把“设备转移（device transfer）”流程产品化**：你后续售后最常见的就是“换手机了/刷机了”。
- **试用上云**：试用由服务端判定并签发结果，客户端只展示/执行限制。

参考：
- Cryptlex Node-locked 介绍：https://cryptlex.com/node-locked-licenses  
- LexActivator（SDK/接口文档入口）：https://docs.cryptlex.com/lexactivator/  
- Verified Trials（试用相关）：https://docs.cryptlex.com/verified-trials  
- Device Transfers（设备转移/换机）：https://docs.cryptlex.com/lexactivator/device-transfers  

## 6. 案例 3：LicenseSpring（强调设备限制与激活/转移）

### 6.1 这个案例的核心做法

- 提供 License API 来做激活、验证、设备转移等能力。
- 典型理念与上面类似：**服务器掌控授权状态**，客户端通过 SDK/HTTP 调用拿到“是否允许用”的结论。

### 6.2 借鉴点

- **把“转移/解绑”做成明确接口与配额**，避免你每次都要人工清表。
- **可在后台查看激活记录**，便于处理纠纷（倒卖/多设备共用）。

参考：
- License API 概览：https://licensespring.com/docs/api/license-api/  
- Activation and Device Transfers：https://licensespring.com/docs/api/license-api/activation-and-device-transfers/  

## 7. 案例 4：Lemon Squeezy（电商卖 key + 授权 API）怎么借鉴

### 7.1 核心做法

- 把“支付后生成 license key、对外提供激活/验证接口”打包成一套。

### 7.2 对你私域售卖的启发

即使你不使用它们的支付体系，也可以学习它们的产品化设计：

- 订单（卖货）与授权（license）分离：订单只是来源，**授权状态应该独立可撤销/可续费**。
- 设计“实例（instance）/设备绑定”的概念，限制同一 key 多端使用。

参考：
- License Keys + Licensing API：https://docs.lemonsqueezy.com/help/licensing/license-keys  

## 8. 案例 5：SDK 厂商的 License Key（Genius Scan SDK / Apryse）常见绑定方式

这类案例不是“给终端用户的激活码”，而是“给开发者/应用授权”的 license key，但非常值得借鉴其思路：

- **License key 通常绑定到应用标识（package name / application ID）并可能含到期时间**。
- Android 里常见注入方式：
  - 初始化时设置 license key
  - 或在 `AndroidManifest.xml` 里以 meta-data 的方式配置

借鉴点（用到你这里就是：把授权绑定到“App 身份”与“设备身份”两层）：

- 服务端绑定时，把 `device_id` 与（`package_name` + 证书指纹）一起纳入校验，降低“二次打包/改包继续用”的概率。

参考：
- Genius Scan SDK：License key 与应用/到期的说明：https://geniusscansdk.com/documentation/guide/installation/license_key  
- Apryse（PDFTron）Android：Manifest 配置 license key 示例：https://docs.apryse.com/documentation/mobile/guides/licensing/adding-license-key/android/  

## 9. 案例 6：Cloudflare Workers + KV / D1 + Rate Limiting（自建 0 成本优先模板）

这是最贴合你“私域售卖 + 低成本 + 易管理”的落地方式。

### 9.1 组件如何对应你的需求

- Workers：承载 API（`/trial/start`、`/license/activate`、`/auth/refresh` 等）
- D1：存结构化数据（license、绑定设备、到期时间、换机次数、状态）
- KV：存缓存/计数（例如限流计数器、近期刷新结果缓存）
- Rate Limiting：保护 API 不被暴力猜码/刷接口

> 具体免费额度会变化，建议以官方文档为准；本节只讨论“组件是否支持该能力”。

参考：
- Workers KV 概览：KV 是分布式 key-value 存储：https://developers.cloudflare.com/kv/  
- D1 概览：D1 是 serverless SQL 数据库：https://developers.cloudflare.com/d1/  
- Cloudflare Rate Limiting（API 文档入口）：https://developers.cloudflare.com/api/resources/rate_limits/  

### 9.2 最小可行 API（MVP）建议

只做 4 个端点就能跑通私域售卖：

1. `POST /v1/trial/start`：创建/返回试用状态（按 device_id 记账）
2. `POST /v1/license/activate`：激活码 + device_id → 绑定授权 → 下发 token
3. `POST /v1/auth/refresh`：token + device_id → 续期/刷新 token（实现离线宽限）
4. `POST /v1/admin/code/create`：后台生成激活码（带期限/换机次数/商品类型）

### 9.3 后台管理怎么做到“最省事”

不一定要做网页后台。你可以先做：

- 一个“管理脚本/管理页面”：
  - 输入：订单号、设备码、商品类型、有效期
  - 输出：激活码（复制发给买家即可）
- 一个“查询接口”：
  - 按激活码/设备码查状态（是否绑定、到期、剩余换机次数）

## 10. 案例 7：Play Integrity API（只做“完整性加分项”，不做私域授权主方案）

Play Integrity 更适合“Google Play 分发 + 购买授权”的场景：

- 它提供多类完整性/许可相关的 verdict（需要服务端验证回执）。
- 其中的 **app licensing verdict** 与 Google Play 购买/安装链路相关；私域 sideload 分发通常无法依赖它来做“是否付费”判断。

但它仍可能作为风控信号用于：

- 判断“是否疑似被二次打包/篡改”
- 对高风险设备提高校验频率或缩短离线宽限

参考：
- Play Integrity 概览：https://developer.android.com/google/play/integrity  
- Play Integrity：Licensing verdict：https://developer.android.com/google/play/integrity/licensing  
- Android Developers Blog（关于“未通过 Play 安装的 app”与信号差异的说明）：https://android-developers.googleblog.com/2022/06/play-integrity-api-update.html  

## 11. 面向你项目的推荐组合（不需要你现在就拍板细节）

给你一个“默认可落地、售后少、成本低”的推荐组合（之后你随时可改策略）：

1. **主授权**：自建 Workers + D1（随机激活码 + 设备绑定 + 可撤销）
2. **试用**：服务端记账（按次数或按时间，二选一）
3. **离线宽限**：先设 72 小时（售后更少），观察滥用情况后再收紧到 24h 或 0h
4. **换机策略**：默认 1 次免费换机（写进 license 记录里）
5. **可选风控**：Play Integrity（只做“提高门槛/风险评分”，不做授权真相来源）

## 12. 你还没想好的 4 个点：怎么用“案例结论”来拍板

1. **离线宽限要不要**  
   - 想要“更安全” → 0h（每次启动必须联网）  
   - 想要“售后少” → 24–72h（多数商业产品都会给宽限期/缓存票据）
2. **试用按时间 vs 按次数**  
   - 更抗“改时间” → 按次数  
   - 更符合用户直觉 → 按时间
3. **1 码 1 设备 vs 1 码 2 设备**  
   - 更防共享倒卖 → 1 设备（推荐）  
   - 更少“家里两台设备想用”的投诉 → 2 设备（风险更高）
4. **换机/解绑策略**  
   - 默认给 1 次，超出人工处理（可收服务费/验证订单）

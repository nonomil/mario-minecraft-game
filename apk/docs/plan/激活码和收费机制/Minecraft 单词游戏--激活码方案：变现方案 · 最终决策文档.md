# Minecraft 单词游戏--激活码方案：变现方案 · 最终决策文档

> 适用场景：个人开发者，目标用户为小学生家长，分发渠道为小红书/咸鱼，开发时间预算 1-2 周。
>  本文综合三份调研文档，给出「推荐路线 + 备选路线 + 落地步骤」的完整决策。

------

## 0. 一句话结论

**首选微信小程序 + 内购买断**；如果你想立刻验证付费意愿、暂时不想做小程序，可以先走 **APK + Cloudflare 激活码** 过渡，后续再迁移。

------

## 1. 为什么不推荐「只卖 APK」

| 风险点       | 具体表现                                                     |
| ------------ | ------------------------------------------------------------ |
| 安装门槛高   | 部分手机反诈拦截，家长看到"未知来源"直接放弃                 |
| 客诉率高     | 换手机/刷机后无法用、孩子误删、激活码输错                    |
| 传播链路断   | 家长群无法一键转发，扫码即玩才是病毒传播的前提               |
| 防倒卖成本高 | 需要自建 License Server，持续维护，且 APK 被逆向后仍有被破解风险 |

------

## 2. 推荐方案：微信小程序 + 内购买断

### 2.1 为什么适合你

- 目标用户是小学生家长 → 决策链是「家长群/小红书看到 → 微信扫码 → 让孩子玩」
- 微信账号天然绑定，一个账号买断，无法转让，**不需要额外防倒卖系统**
- 微信处理支付与账号，你几乎零售后
- 游戏已经是 Web 实现，移植成本相对低

### 2.2 付费墙设计（建议）

```
免费内容（让用户传播）
├── 基础英语词库（约 200 词）
├── 1 个存档槽
└── 全部游戏机制可体验

付费内容（建议定价 9.9 ~ 19.9 元买断）
├── 全词库：英语 / 语文 / 幼小衔接
├── 自定义词库导入（.txt / .csv）
├── 全存档槽 + 进度云同步
└── 去广告（可选）
```

### 2.3 技术风险：必须提前验证

你的游戏依赖 **摄像头 + 前端 AI 姿态识别（MediaPipe）**，微信小程序的 WebView 在以下两点存在风险：

- WebGL 支持程度弱于原生 Chrome/Safari
- 算力限制可能导致帧率下降

**建议在动手迁移前，先用微信内置浏览器打开现有网页版，测试帧率与摄像头权限是否正常。** 若流畅度不达标，参考备选方案 B。

### 2.4 1-2 周落地步骤

**第 1 周：迁移与基础功能**

1. 注册微信小程序（个人主体，教育类目，年费 300 元）
2. 使用 `uni-app` 框架，复用现有 JS 游戏逻辑
3. 将 `localStorage` 替换为 `wx.setStorageSync`
4. 触屏按键已存在 ✅，重点调试摄像头权限与帧率

**第 2 周：付费与上线**

1. 接入微信小程序内购（虚拟商品，微信抽成约 6%）
2. 实现付费墙：未购买时屏蔽高级词库入口
3. 内测 10 位家长，收集反馈
4. 发第一条小红书推广内容（见第 5 章）

## 3. 备选方案 A：APK + Cloudflare 激活码（适合「先验证付费意愿」）

如果你不想立刻做小程序，可以先用这套方案在咸鱼/小红书验证用户愿不愿意付费，再决定是否迁移到小程序。

### 3.1 基础设施（真正 0 成本）

| 组件               | 用途                                         |
| ------------------ | -------------------------------------------- |
| Cloudflare Workers | 承载 API（试用/激活/刷新）                   |
| Cloudflare D1      | 存储 license、设备绑定、状态（免费额度够用） |
| Cloudflare KV      | 限流计数器、token 缓存                       |

### 3.2 最小 API（4 个端点就能跑通）

```
POST /v1/trial/start        # 按 device_id 记账，防重装无限试用
POST /v1/license/activate   # 激活码 + device_id → 绑定 → 下发 token
POST /v1/auth/refresh       # token 续期，实现离线宽限
POST /v1/admin/code/create  # 后台生成激活码（带管理口令保护）
```

### 3.3 激活码设计

- 格式：`XXXX-XXXX-XXXX-XXXX`（Base32，去除 `0 O 1 I` 易混淆字符）
- 服务端只存 `hash(code)`，校验时对比哈希
- **不做客户端算法推导**——一旦被逆向会出现万能码生成器

### 3.4 推荐默认策略

| 策略项   | 推荐值                      | 原因                           |
| -------- | --------------------------- | ------------------------------ |
| 试用形式 | 按次数（10 次启动）         | 防修改系统时间；比"按天数"更稳 |
| 设备绑定 | 1 码 1 设备                 | 防共享倒卖                     |
| 换机次数 | 默认 1 次免费，超出联系客服 | 平衡售后成本                   |
| 离线宽限 | 72 小时                     | 国内网络波动；减少差评与退货   |
| 授权类型 | 先只做永久买断              | 运营最简单                     |

### 3.5 设备标识（device_id）

```
ANDROID_ID → SHA-256 哈希 → 混入 packageName + 证书指纹 → 再次哈希
```

- 只传哈希值，不传原始标识，保护隐私
- 同时绑定 App 身份，降低"二次打包"风险

### 3.6 数据模型（最小 3 张表）

```sql
-- devices
device_id TEXT PRIMARY KEY, first_seen_at, last_seen_at, banned BOOLEAN

-- trial_state
device_id TEXT, remaining_runs INTEGER, consumed_at TIMESTAMP

-- licenses
license_id TEXT, activation_code_hash TEXT, status TEXT,
bind_device_id TEXT, expires_at TIMESTAMP,
rebind_quota INTEGER, rebind_used INTEGER,
created_at, activated_at
```

### 3.7 现实局限（必须承认）

- APK 被转发后必然被研究，无法「绝对防破解」，只能「显著抬高门槛」
- 售后（换机/激活失败/网络问题）会持续消耗你的时间
- 客单价在私域通常不超过 30 元，ROI 不如小程序

## 4. 备选方案 B：网页版 + 广告/打赏（0 开发成本，立刻可用）

如果小程序帧率测试不通过，或者你想先「零成本启动」：

- 现有网页版接入 **Google AdSense**（游戏间隙展示广告）
- 底部放微信/支付宝**打赏二维码**
- 付费词库用「微信收款 → 手动发解锁码」实现（最简单，初期够用）

这套方案今天就能上线，适合先测市场反应。

## 5. 冷启动：小红书推广思路

小学生家长在小红书上的决策路径是：**看到视频 → 觉得有趣 → 转给家长群**。

**内容方向（按效果排序）**

1. **实拍孩子玩游戏的视频**：孩子跳来跳去学单词，家长群天然传播
2. **"自己做了个游戏"的开发故事帖**：个人开发者人设，增加信任感
3. **词库截图 + 功能说明**：针对有需求的家长做决策参考

**标题关键词**：`小学英语` / `单词记忆游戏` / `体感游戏` / `我的世界` / `孩子自己玩`

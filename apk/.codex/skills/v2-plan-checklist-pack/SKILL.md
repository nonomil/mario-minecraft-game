# v2-plan-checklist-pack

这个 skill 用来把“口头讨论过的方向”固定成可执行文档，避免每次重新解释边界、优先级和已完成状态。

## 何时使用

- 用户说“重新梳理计划”“给我总计划和分计划”“生成可打钩文档”
- 准备新开 worktree，但需要先冻结范围和步骤
- 旧计划已经失真，需要基于最近对话重新整理
- 需要把“已完成 / 未完成 / 风险 / 下一步”做成连续可回填文档

## 固定输出

至少输出 3 层文档：

1. `docs/plans/YYYY-MM-DD-v2-x-<slug>.md`
   - 作为实施主计划
   - 写清目标、范围、任务阶段和验收
2. `docs/PV测试结果对齐分析/NN-V2.x...总计划-打钩版.md`
   - 作为总览入口
   - 写清保留项、排除项、阶段状态
3. `docs/PV测试结果对齐分析/NN-V2.x...子计划-打钩版.md`
   - 至少 2 份
   - 一份给算法与数据流
   - 一份给 UI 与验收闭环

## 推荐结构

### 总计划文档

- 背景
- 第一阶段目标
- 必保留
- 明确排除
- 相关文档
- 当前状态
- 验收标准

### 分计划文档

- 任务分组
- 每组下的 checkbox
- 当前状态
- 风险与注意事项

### 实施主计划

- Current Checklist
- Related Documents
- Scope Rules
- Task 1 ~ Task N
- Acceptance

## 打钩规则

- 只把已经有证据的事项打成 `[x]`
- 文档创建本身可以打钩
- 未开始实现的代码任务保持 `[ ]`
- 如果状态变化，必须同时更新总计划和相关分计划

## 必做检查

1. 文档命名连续、编号清晰
2. 文档之间有交叉链接
3. UTF-8 回读成功
4. 用 `unicode_escape` 抽查前 5-20 行，确认没有连续问号污染

## 不要做的事

- 不要只写一份泛泛的长文
- 不要把旧计划整段复制过来不修状态
- 不要把“还没做的代码工作”提前打钩

## 仓库内推荐触发话术

- “用 `v2-plan-checklist-pack` 重新梳理总计划和分计划”
- “把最近对话整理成打钩文档”
- “给 `V2.2` 出总计划 + 子计划 + 步骤计划”

# 学习报告优先：最小执行路线与实现计划（2026-03-16）

> 背景：你已获得“小范围孩子愿意玩”的真实反馈，但家长反馈分化（有的觉得好、有的没感觉）。  
> 关键结论：问题通常不在“游戏好不好玩”，而在**家长感知不到学习效果**。  
> 本文目标：把“接下来到底做什么”压缩到最少步骤，并给出学习报告（Learning Report）的可落地实现计划；变现/激活码先降级为“后置选项”。

---

## 0. 一句话决策

**先做「学习效果可视化」再做「变现系统」**：最小改动是增加一个“今日学习报告/学习统计”页面，并在小红书视频里重点展示它。

---

## 1. 为什么学习报告优先级高（来自真实反馈）

出现场景（教育类游戏最常见）：

- 孩子玩得开心  
- 家长看不出学了什么  
- 家长结论：这只是游戏 → 付费动力弱

相反，如果家长能一眼看到：

- 今天遇见了多少词 / 正确率 / 玩了多久  
- 哪些词答错了（“还需练习”）  
- 连续学习天数（激励与打卡）

家长会更容易把它当作“学习工具”，从而形成付费动机。

来源笔记：`学习报告记录.md`、`小红书推广.md`。

---

## 2. 你现在只需要做两件事（按顺序）

### 2.1 第 1 件：加「学习报告」MVP（1–2 天，纯前端/本地）

目标不是“做得很全”，而是让家长能截图/转发，并且看得懂。

### 2.2 第 2 件：发小红书（视频必须展示学习报告）

原策略是“先发链接验证有没有人玩”；现在升级为：

- 仍然发链接（不改分发链路）  
- 但视频里专门展示学习报告（让家长第一眼就懂“这是学习工具”）

> 小红书文案模板见：`小红书推广.md`（建议保持可对比）。

---

## 3. 学习报告 MVP：只保留家长最关心的 3 个数字 + 1 个动作

### 3.1 页面内容（建议固定布局）

**顶部（家长 3 秒看完）：**

1. `今日遇见词`：今日 unique 词数（不是词库总量）  
2. `今日正确率`：挑战题正确率（success / (success+fail)）  
3. `今日时长`：今日游玩分钟数

**中部（让家长产生动作）：**

- `还需练习`：把今日答错的词单独列出来（最多 5–10 个），支持点击“再听一遍/再看一遍”

**底部（激励孩子 + 传播）：**

- `连续学习天数`：≥1 个学习事件/天 即算学习
- `截图分享`按钮：把报告页转成图片（或生成可保存的长图）

### 3.2 一条底线：报告页必须“可截图解释”

也就是：家长把手机递给你问“学了什么”，孩子/家长能指着页面回答，不需要你解释规则。

## 4. 现有代码基础：并不是从 0 开始

### 4.1 已存在的数据与事件（可直接复用）

- 学习事件记录（带时间戳）：`src/modules/09-vocab.js` 内 `recordLearningEvent()`  
  - 事件包含：`source`（vocab/challenge/village）、`result`（success/fail/partial）、`ts`  
- 词汇采集与挑战结果写入 pack 进度：`src/modules/12-challenges.js`  
  - `recordWordProgress()`（采集词）  
  - `writeChallengeResultToProgress()`（correct/wrong 计数 + lastSeen）  
  - `_sessionWordResults`（本局题目对错记录）
- 个人档案弹窗里已有“学习统计面板”：`src/modules/08-account.js`  
  - `renderLearningStats()`（趋势/弱词/正确率等）  
  - 但目前依赖 `account.vocabulary.dailyStats / wordStats`，这些字段**尚未真正写入**（见 4.2）。

### 4.2 目前的缺口（导致“家长没感觉”）

- `account.vocabulary.wordStats` 与 `account.vocabulary.dailyStats` 在 UI 里被读取，但仓库里基本没有写入逻辑  
  - 结果：学习统计面板可能展示不完整或为空
- 游戏结束页目前展示“学习单词数”，但 `getLearnedWordCount()` 统计口径可能不符合“家长直觉”（需要改成“今日遇见词/本局遇见词”）

> 这意味着：学习报告要做得像样，关键不在 UI 花哨，而在把“可解释的统计口径”写进存储。

## 5. 推荐落地方案（最小改动、可持续扩展）

### 5.1 数据策略（推荐）

**新增一个“按日聚合”的本地统计结构**，每次学习事件发生就更新一次。

理由：

- 不依赖后端  
- 不依赖“最近事件列表长度限制”（`recentEvents` 只有 100 条）  
- 能直接支撑“今日报告/连续天数/弱词”

建议存储位置：

- 如果玩家已登录档案：写入 `currentAccount.vocabulary`（随 `saveCurrentProgress()` 一起持久化）  
- 如果未登录：先写入 `progress` 或单独 localStorage key（待创建档案时合并）

建议最小字段（够用就行）：

```js
account.vocabulary.daily = {
  version: 1,
  days: {
    "2026-03-16": {
      playSeconds: 0,
      words: {
        "apple": { seen: 3, correct: 2, wrong: 1, lastTs: 0, zh: "苹果" }
      }
    }
  },
  streak: { current: 0, lastDay: "2026-03-16" }
}
```

> 注意：这不是要替代 packProgress，而是为“家长可读报告”单独建一个轻量索引。

### 5.2 UI 策略（推荐）

做一个独立弹窗/页面：**今日学习报告**。

入口建议（按影响排序）：

1. **游戏结束（gameover overlay）**：新增按钮“📊 今日学习报告”（家长最可能看到）  
2. 首页/开始页：新增“学习报告”入口（可选）  
3. 个人档案弹窗：保留现有 `learning-stats-panel`，后续可复用同一套渲染函数

截图分享：

- Web：优先 Web Share API（支持则直接分享图片），不支持则提供“下载图片”  
- APK（WebView/Capacitor）：可后续接 Capacitor Share（不作为 MVP 必做）

## 6. 验收标准（你自己 3 分钟就能验）

1. 新装打开，玩 1–2 分钟，结束后能点开“今日学习报告”
2. 报告页至少显示：
   - 今日遇见词（>0）
   - 今日正确率（如果做过挑战题，能看到百分比）
   - 今日时长（>0）
   - 还需练习（做错过题时能看到词条）
3. 刷新页面/重开后数据仍在
4. 连续两天各玩 1 次，连续学习天数 +1

## 7. 详细实现计划（不马上开工，先定“怎么做”）

> 说明：该实现会涉及多文件（`Game.html`、`src/modules/08-account.js`、`src/modules/10-ui.js`、可能还有 `src/styles/*`），属于“跨模块改动”。按协作规范，进入编码前建议你先确认一次“是否按此方案落地”。

### Step A：补齐数据记录（核心）

1. 定义 `getLocalDayKey()`：本地时区 YYYY-MM-DD  
2. 在学习事件发生处更新“今日聚合”
   - vocab 采集（`recordWordProgress()` 已有事件）  
   - challenge 成功/失败（`completeLearningChallenge()` 已有事件）  
3. 每次更新后写入存储（已登录写 account；未登录写 progress）
4. 增加 streak 计算：如果今天第一次学习且昨天也学习过 → streak+1，否则重置为 1

### Step B：做报告页 UI（家长可读）

1. 在 `Game.html` 增加一个 modal 容器（结构类似 profile-modal）  
2. 在 `src/modules/08-account.js` 增加 `renderTodayLearningReport()`：生成 HTML  
3. 在 `src/modules/10-ui.js` 或 UI 模块里加打开/关闭逻辑 + 按钮事件绑定  
4. 样式：复用现有 `.profile-modal` / `.profile-stats`，新增少量卡片样式即可

### Step C：入口放在“游戏结束页”（转化关键）

1. gameover overlay 增加按钮“📊 今日学习报告”  
2. 若无学习数据：显示空态文案（提示多玩几分钟/完成挑战）

### Step D：可选（但强烈建议）加 1 条 E2E

用 Playwright 写一条最小用例：

- 打开页面 → 触发一次学习事件（如果测试环境可控）→ 打开报告 modal → 断言出现“今日学习报告”标题与某个数字字段

命令：`npm run test:e2e`

- **没人玩**：先优化传播/定位（不是激活码能解决的）

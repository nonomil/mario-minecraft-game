- 版本目标：用最多并行线抢时间
- 并行判断：`可以并行，但只适合多人协作且愿意承担高合并成本`
- 推荐级别：`备选，不作为首选`

## 为什么能并行

- 骑龙线与村庄线共享代码最少
- 地下矿洞线可以先做配置、渲染和敌人草案
- 坚守者蛋线可以先做数据和召唤骨架

## 为什么风险最高

- `Game.html` 会被 C 和 A 同时改
- `src/modules/16-events.js` 会被 B/C 同时改
- `src/modules/18-village.js` 会被 A/E 同时改
- `config/game.json`、`config/biomes.json`、掉落与敌人逻辑会在 D/E 交叉
- 最后集成分支会吃下几乎所有冲突

## 适合的协作前提

- [ ] 至少 3 人以上开发
- [ ] 每条流都能独立维护测试
- [ ] 每天固定一次接口同步
- [ ] 有专门集成 owner

## 全并行版管理要求

- [ ] 每条流先写接口说明，不允许先各写各的
- [ ] 每条流必须写“改动文件清单”
- [ ] 每条流每日同步一次状态
- [ ] 每次合并后立即跑语法检查和相关测试
- [ ] 集成分支不做新功能，只做合并与修复

## 结论

- 如果当前目标是“尽快开始并保持稳定”，不要直接选 V3
- 如果后续确认有多人同时写代码，且愿意安排专门集成人员，再考虑从 V2 升级到 V3

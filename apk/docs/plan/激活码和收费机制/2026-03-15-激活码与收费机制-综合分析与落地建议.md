## 1. 先定路线：你真正要卖什么（两条主路线）

### 路线 A：微信小程序 + 买断（对“家长群扫码即玩”更友好）

- **收费机制**：买断（9.9~19.9 这种低客单更匹配私域冷启动），可选广告/去广告
- **为什么省售后**：身份=微信账号（openid），无需“激活码 + 设备码 + 换机”这一套
- **技术风险**：WebGL/摄像头/算力（MediaPipe 等）在小程序环境要先验证

> 结论：如果你目标用户主要是“小学生家长 + 群传播”，小程序通常是终局；但需要迁移验证性能与权限。

### 路线 B：APK 私域售卖 + 在线激活码（更贴合“咸鱼成交即发码”）

- **收费机制**：你收款 → 你发激活码 → App 联网绑定设备 → 按权益放行
- **优点**：不必迁移到小程序；能快速验证“是否有人愿意付费”
- **成本/售后现实**：必须处理换机/激活失败/网络失败；因此授权策略要“默认抗售后”

> 结论：APK + License Server 适合作为“过渡/验证”或长期私域渠道；但要把策略做得足够省事。

## 2. 收费机制怎么设计才“最便宜 + 好管理”

私域售卖的核心不是做 10 个 SKU，而是：**SKU 少、规则清晰、售后少、可解释**。

### 2.1 推荐最小 SKU（从 1 个起步）

**SKU-1（最建议先上）：买断-单设备**
- 1 个激活码绑定 1 台设备
- 永久可用（`expires_at = null`）
- 免费换机 1 次（`rebind_quota = 1`）
- 离线宽限 72h（见 3.2）

> 这一个 SKU 就能跑通“试用 → 转化 → 激活 → 售后”全链路。

### 2.2 可选增收 SKU（等你验证付费意愿后再加）

**SKU-2：年卡（单设备）**
- `expires_at = now + 365d`（到期需续费）
- 优点：现金流更稳定；缺点：续费触达与售后会增加

**SKU-3：家庭版（双设备）**
- `device_limit = 2` 或 2 次绑定额度
- 优点：减少“家里两台设备”投诉；缺点：更容易共享/倒卖

### 2.3 “买断 + 续更（升级维护）”是低售后的一种折中

很多授权产品把“能不能用”和“能不能用最新版”分开：

- `expires_at`：使用权（永久/到期）
- `update_until`：更新权（例如购买后免费更新 30 天/365 天）

落地方式（对私域 APK 也适用）：
- 老用户即使不续更，也能继续使用**旧版本**；但安装新版本时，服务端按 `update_until` 决定是否放行新版本权益
- 优点：不逼用户订阅；缺点：需要你在服务端对 `app_version/build` 做策略判断

## 3. 激活码与授权策略（把“收费机制”映射成服务端字段）

### 3.1 推荐的数据字段（足够覆盖上面的 SKU）

对每个 license（授权）建议最少这些字段：

- `status`: active / revoked / frozen
- `bind_device_id`: 绑定设备（或维护 `devices[]` 列表）
- `device_limit`: 1（默认）或 2（家庭版）
- `expires_at`: 空=永久；非空=年卡到期
- `update_until`: 空=不限制更新；或按“续更”控制新版本权益
- `rebind_quota`, `rebind_used`: 换机次数
- `entitlements`: 功能开关（词库、存档槽、去广告、导入等）

### 3.2 推荐默认策略（以“少售后”为第一目标）

- **离线宽限**：72h（token 有效期/宽限期一致，网络抖动不致死）
- **试用**：按次数（例如 20 次启动）比按时间更抗“改系统时间”
- **绑定**：1 码 1 设备；默认允许 1 次换机；异常频繁换机自动冻结
- **激活码**：随机高熵；服务端只存 `hash(code)`；不要做客户端可推导算法码

### 3.3 关键安全线（一定要守）

- 客户端 **不能**内置任何“管理密钥/后台 token”
- 激活/刷新必须走服务端；服务端做限流与日志；否则很快会出现“猜码/刷接口/自动化攻击”

## 4. 私域“收款 → 发码 → 售后”的最低管理成本流程

### 4.1 人工发码（最贴合咸鱼/小红书，成本最低）

建议你固定一个 SOP：

1. 用户付款（订单号/截图）  
2. App 内复制“设备码/设备指纹”发给你  
3. 你在后台生成激活码（选择 SKU：买断/年卡/家庭版）  
4. 用户输入激活码完成绑定  
5. 台账记录（CSV/Excel）：订单号、激活码、device_id、SKU、到期、换机次数

### 4.2 半自动发码（当你单量上来再做）

两种常见模式（不强制）：

- **电商 + 自动发 key**：如 Gumroad/Lemon Squeezy 等可卖“license key”，再由你自己的服务端做激活与设备绑定
- **第三方 License 平台**：Keygen/Cryptlex/LicenseSpring 等，直接提供 license+设备激活+撤销（通常付费）

> 注意：这些平台的支付/结算/合规与国内私域链路未必匹配；对你来说更多是“学习其流程设计”，不一定要用。

## 5. 退款/纠纷/封禁：提前写清楚能省大量售后

建议你把规则写进 App 激活页与购买说明（越短越好）：

- 试用可用多少次/多久
- 一个码默认绑定 1 台设备；可免费换机 1 次（或联系客服）
- 退款政策：建议“未激活可退；已激活需协商”（按你实际情况）
- 封禁政策：明确“倒卖/共享/异常换绑会冻结，需要人工核验订单解封”

## 6. 落地优先级（1~2 周现实路线）

> 你现在“还没想好”很正常，建议用最小投入把数据跑出来。

1. **先落地 APK 授权 MVP（4 个端点）**：试用 / 激活 / 刷新 / 管理端发码  
2. **只上 1 个 SKU（买断-单设备）**：把售后磨顺（换机、离线宽限、错误提示）  
3. **观察 1–2 周数据**：转化率、激活失败率、换机率、共享倒卖率  
4. 再决定是否迁移小程序（如果传播需求强、用户是家长群）

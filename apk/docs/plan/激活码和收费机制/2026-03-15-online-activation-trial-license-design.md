# 私域分发 APK：联网试用 + 在线激活（License Server）方案讨论稿（Design）

> 适用场景：小红书/咸鱼等私域渠道售卖 APK，通过“试用 → 购买 → 激活码授权”的方式控制转发/倒卖与售后成本。  
> 本文是讨论稿：用于把需求、约束、方案、风险与待决策项一次性整理，便于后续落地到本仓库。

## 0. 结论（推荐默认方案）

- **采用在线授权（license server）**：客户端每次启动（或在离线宽限期内）向服务端校验授权。
- **试用也走服务端记账**：试用额度按 `device_id` 记录，避免“卸载重装无限试用”。
- **激活码人工发放**：你在后台/脚本生成激活码（或后台创建授权并给出激活码），买家在 App 内输入完成绑定。
- **默认授权策略（建议）**
  - 设备绑定：**1 个激活码 = 1 台设备**（可配置 1–2 次换机/重置绑定次数）。
  - 离线宽限：**允许离线 24–72 小时**（减少网络波动带来的差评与售后）；超过必须联网续期。
  - Token：短有效期（例如 24–72 小时）+ 服务端可撤销/拉黑。

## 1. 目标与非目标

### 1.1 目标（你真正想解决的问题）

1. **防转发/倒卖**：APK 被转发后不能“装上就永久免费”；激活码也不能一人购买多人共用。
2. **试用可控**：用户能先体验，但不能无限重装刷试用。
3. **成本最低 & 易管理**：不买昂贵系统、不做复杂账号体系；你发码/禁用/换机都能低成本处理。
4. **售后压力可控**：尽量减少“忘账号/收不到验证码/换机复杂”等售后。

### 1.2 非目标（避免不切实际）

- **不追求强对抗破解**：只要客户端可被逆向，就无法“绝对防破解”。本方案追求的是“显著抬高门槛 + 运营可控”。
- **不做复杂支付体系**：私域售卖通常不需要内购/订阅平台；如未来上架再单独规划。

## 2. 约束与现实边界

### 2.1 约束

- 客户端 **可以且要求联网**（你已确认）。
- 分发渠道为私域（小红书/咸鱼），通常是“你收款 → 你发激活码/授权”。
- 需要兼顾国内网络环境的波动与可用性（因此建议离线宽限）。

### 2.2 现实边界（必须提前承认）

- **只要 APK 泄露，就会被研究**：所以任何“客户端内置密钥/简单算法码”都容易被做出万能码。
- **服务端是可控性的来源**：撤销、封禁、换机次数、并发限制、试用记账，都需要服务端参与。
- **“必须全程联网”更安全，但用户体验更差**：因此推荐“短离线宽限 + 强制定期联网”折中。

## 3. 方案对比（为什么推荐在线授权）

### 3.1 纯离线签名激活码（不推荐作为主方案）

优点：0 服务器、离线可用、发码简单。  
缺点：无法撤销/封禁；试用难防重装；一旦出现“破解器”你很难止损。

### 3.2 在线授权（推荐）

优点：
- 可撤销、可封禁、可限制设备数、可记录试用与激活历史；
- 激活码泄露也可快速作废；
- 可以做“换机/重置绑定”策略，控制售后成本。

缺点：
- 需要一个轻量服务（但可选 0 成本免费额度平台）；
- 需要处理网络失败的用户体验（因此建议离线宽限）。

## 4. 用户流程（User Journey）

### 4.1 试用流程（必须联网）

1. App 启动 → 生成 `device_id`（见第 6 章）  
2. 调用服务端：`POST /v1/trial/start`  
3. 服务端返回试用状态：
   - `trial_active=true` 且 `trial_expires_at=...` 或 `trial_remaining_runs=...`
   - 或 `trial_available=false`（已用完/已激活）
4. 客户端按返回的权益决定是否放行/限制功能。

### 4.2 购买与激活流程（人工售卖）

1. 用户点击“激活/购买” → App 展示：
   - 设备码（可一键复制）
   - 购买指引（联系渠道/订单截图/售后说明）
2. 你在后台/脚本生成激活码（可绑定设备，也可先不绑定）  
3. 用户输入激活码 → `POST /v1/license/activate`
4. 服务端完成绑定并返回 `access_token` 与 `entitlement`（权益）。

### 4.3 日常校验流程（在线 + 离线宽限）

推荐策略：
- 启动时尝试刷新：`POST /v1/auth/refresh`
- 若网络失败：
  - 若本地 `access_token` 未过期：允许进入（显示“离线宽限剩余时间”）
  - 若已过期：阻止进入并提示“需要联网验证授权”

## 5. 授权策略（Policy）

### 5.1 设备绑定策略（强烈建议）

- 默认：**1 个激活码只能绑定 1 台设备**
- 允许换机：每个 license 提供 `rebind_quota`（例如 1–2 次）
- 如果发现异常（同一 license 多设备/频繁换绑），可自动冻结并进入人工处理。

### 5.2 试用策略（服务端记账）

试用建议用两种之一（任选其一，避免同时叠加导致体验混乱）：
- **按时间**：例如 3 天/7 天；到期后需要激活
- **按次数**：例如 20 次启动/10 局游戏；更抗“改时间”

> 备注：私域售卖里，“按次数”通常更省售后（用户不会因为手机时间设置奇怪而误伤）。

### 5.3 授权类型（商品形态）

为便于你运营，建议先支持两种：
- `PERPETUAL`：永久（一次性售卖）
- `TIME_LIMITED`：按时长（30 天/365 天），更适合后续做持续收入

### 5.4 离线宽限（建议默认 72h 上限）

建议默认：**离线最多 24–72 小时**。  
原因：国内网络不稳定；完全强制在线会导致差评/退货/售后激增。

## 6. 设备标识（Device ID）设计

目标：同一设备稳定、跨卸载尽量稳定、且尽量不收集隐私。

### 6.1 推荐做法（Android APK）

- 取 Android 端可得的稳定标识（例如 `ANDROID_ID`）→ 做哈希（SHA-256）
- 混入 App 身份信息（例如 `packageName` + 证书指纹/签名指纹）再哈希一次
- 最终得到 `device_id`（只存 hash，不存原始标识）

### 6.2 注意事项

- 不建议用“可变的硬件信息”当唯一标识（容易因系统升级/权限变化导致设备识别漂移）。
- 如果项目是 WebView/PWA 容器：需要通过 native bridge 把 `device_id` 提供给 JS 层使用。

## 7. 服务端接口草案（API Draft）

统一前缀：`/v1`

### 7.1 试用

- `POST /trial/start`
  - 输入：`device_id`, `app_version`, `build`, `platform`
  - 输出：`trial_active`, `trial_expires_at` 或 `trial_remaining_runs`

### 7.2 激活

- `POST /license/activate`
  - 输入：`device_id`, `activation_code`
  - 输出：`access_token`, `expires_at`, `entitlement`

### 7.3 刷新与权益查询

- `POST /auth/refresh`
  - 输入：`device_id`, `access_token`
  - 输出：新的 `access_token` 与 `entitlement`
- `GET /entitlement`
  - 输入：`access_token`
  - 输出：`entitlement`

### 7.4 后台管理（最小集合）

> 后台不一定需要做复杂网页；也可以是“脚本 + 管理口令”。

- `POST /admin/code/create`：生成激活码（可指定期限/类型/换机次数）
- `POST /admin/license/revoke`：撤销某 license
- `POST /admin/device/ban`：封禁 device
- `GET /admin/license/query`：按 code/license_id 查询状态

## 8. 数据模型草案（DB Schema Draft）

最小需要三张表（可加索引）：

1. `devices`
   - `device_id`（主键）
   - `first_seen_at`, `last_seen_at`
   - `banned`（bool）
2. `trial_state`
   - `device_id`（外键）
   - `trial_expires_at` / `remaining_runs`
   - `consumed_at`
3. `licenses`
   - `license_id`
   - `activation_code_hash`（只存 hash，避免泄露）
   - `status`（active/revoked）
   - `bind_device_id`（绑定的 device）
   - `expires_at`（可空=永久）
   - `rebind_quota`, `rebind_used`
   - `created_at`, `activated_at`

可选：`events` 表记录关键行为（激活/刷新/失败原因）用于排查售后。

## 9. 激活码设计（Code Format）

### 9.1 推荐：随机码 + 服务端存储（最省心）

- 激活码生成：随机（高熵）字符串
- 格式建议：`XXXX-XXXX-XXXX-XXXX`（便于口述/手输）
- 字符集建议：Base32（去除易混淆字符 `0O1I`）
- 服务端只存 `hash(code)`，校验时 `hash(输入)` 对比

优点：实现简单，可撤销，可统计。  
缺点：需要服务端在线校验（但你已经要求必须联网）。

### 9.2 不推荐：客户端可推导算法码

一旦被逆向就会出现“破解器/发码器”，你很难止损。

## 10. 客户端落地要点（Client Implementation Notes）

### 10.1 存储

- 将 `access_token`、`expires_at`、`last_verified_at` 存在本地（Android 建议加密存储）。
- 每次启动：
  1) 若本地 token 过期 → 必须联网刷新
  2) 若未过期 → 尝试联网刷新（失败则走离线宽限）

### 10.2 UI/文案（降低售后）

- 在“激活页”明确提示：
  - 需要联网
  - 一个码默认绑定一台设备
  - 换机政策（可换几次/需要联系客服）
- 在“失败页”给出可操作指引：
  - 网络失败
  - 激活码错误
  - 已绑定其他设备
  - 授权已撤销/已到期

## 11. 反滥用与风控（轻量即可）

建议先做“够用的”：

- 频率限制：同一 `device_id`、同一 IP 的激活/刷新做 rate limit
- 异常检测：
  - 同一 code 多次失败输入 → 临时冻结该 code
  - 同一 license 在短时间出现多设备尝试 → 冻结并提示人工处理
- 黑名单：封禁 device 或 license

## 12. 部署与成本（0 成本优先）

目标是：**能跑、好维护、免费额度够用**。

候选（不写死额度，避免未来变动）：

- **Cloudflare Workers + D1/KV**：部署简单、免费额度常能覆盖小项目，适合做轻量 API。
- **Supabase / Firebase**：功能齐全，但可能引入更多依赖与控制面板复杂度。
- **自建轻量 VPS**：最自由但需要运维；不是“零成本”。

> 推荐优先级：Cloudflare（轻）→ Supabase（全）→ VPS（重）。

## 13. 待决策项（你需要拍板的 6 件事）

1. **离线宽限**：0 小时（最严）/ 24 小时 / 72 小时（最推荐）
2. **试用形式**：按时间（3/7 天）还是按次数（10/20 次启动）
3. **设备绑定规则**：1 码 1 设备（推荐）还是允许 2 设备（更易被共享）
4. **换机政策**：默认允许换机 1 次还是 2 次；是否需要人工审核
5. **授权形态**：先只做永久，还是同时支持“按时长”
6. **异常处理**：冻结后是“联系客服解封”还是“自动解除”

## 14. 下一步（落地路线）

1. 先按本讨论稿，确定第 13 章的决策项默认值  
2. 选择部署平台（建议 Cloudflare Workers）并确定最小后台管理方式（脚本或网页）  
3. 在客户端接入：
   - 试用校验
   - 激活输入
   - token 刷新与离线宽限
4. 灰度：先给少量用户使用，观察售后与滥用情况，再调策略（例如离线宽限、换机次数）

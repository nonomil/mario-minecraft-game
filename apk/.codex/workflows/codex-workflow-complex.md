# codex-workflow-complex｜复杂开发流程（Codex 主导）

> 本文档遵循 `codex-workflow-constants.md`。
> 触发条件：任意一条不满足简单模式（文件>3 / diff>200 / 需求有歧义 / 跨模块）。
> 入口：从 `CODEX.md` 路由到此。

---

## 强制门禁

1. 先确认需求边界与验收标准
2. 未说“开始开发”前不进入编码
3. 任意阶段发现超范围，立即暂停并回到计划阶段

---

## Phase 0：扫描与路由

- 若命中大型库条件，跳转 `codex-workflow-largebase.md`
- 未命中时做轻量扫描并输出：
  - 目录结构（2 层）
  - 核心模块职责
  - 相关文件与潜在影响点

---

## Phase 1：计划文档

在 `docs/plans/YYYY-MM-DD-[feature].md` 产出：

- 需求理解与边界
- 2-3 个备选方案与取舍
- 涉及文件表（修改/新增/测试）
- 风险与缓解
- 验收标准

---

## Phase 2：工程可行性审查（Claude Code 第二评审）

- Codex 给出计划草稿
- Claude Code 做独立可行性审查，但不接管主流程决策
- Codex 汇总差异并更新计划
- 需用户确认后进入下一阶段

---

## Phase 3：交叉 Review 收敛

每轮执行：

1. Codex 审查（逻辑、边界、兼容）
2. Claude Code 第二评审（风险、遗漏、顺序）
3. Codex 合并结论、做最终仲裁并标记优先级

收敛条件：

- 连续 2 轮无新问题
- 用户确认定稿

---

## Phase 4：任务拆分与并行准备

输出到 `docs/development/`：

- 每个任务单独 steps 文档
- 每任务目标 diff ≤ 200 行
- 并行任务影响范围表（避免文件重叠）

---

## Phase 5：执行与验证

进入执行前必须满足：

- 用户明确“开始开发”
- 任务边界明确

执行中必须满足：

- 单任务仅改授权文件
- 每个任务完成后先测试再提交
- 发现新增范围立即暂停并回到 Phase 4

---

## Review 与止损

- 每任务至少两轮审查（Codex + Claude Code）
- 超过 5 轮仍有新问题：强制拆分任务
- 单任务审查超过 3 小时：暂停并让用户决策

---

## 输出要求

最终对用户输出：

- 本轮完成项
- 变更文件清单
- 验证结果（通过/失败）
- 剩余风险与下一步

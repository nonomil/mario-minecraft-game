# codex-workflow-debug｜Debug 流程（Codex 主导）

> 本文档遵循 `codex-workflow-constants.md`。
> 触发条件：用户描述 bug / 错误信息 / 测试失败 / 意外行为。
> 入口：从 `CODEX.md` 路由到此。

---

## 核心原则

- 先复现后修复
- 先定位根因再改代码
- 修复范围最小化
- 保留可验证证据（测试/日志/命令结果）

---

## Lite 流程（默认）

### Phase 0：Bug 卡片

- 现象
- 复现步骤
- 预期/实际
- 错误栈
- 影响范围
- 严重等级（P0/P1/P2/P3）

### Phase 1：快速排查（≤5 分钟）

- 最近提交是否引入
- 错误是否直接指向文件
- 是否环境/配置问题

能定位则直接到 Phase 4；否则进入 Phase 2。

### Phase 2：根因定位

Codex 输出：

- 根因文件与行号
- 触发条件
- 影响范围
- 置信度（高/中/低）

### Phase 3：修复方案确认

- Codex 给最小修复方案
- Claude Code 做交叉验证（可选但建议）
- 用户确认后进入修复

### Phase 4：实施修复与验证

执行顺序：

1. 写复现测试（先失败）
2. 按方案做最小修复
3. 运行复现测试（应通过）
4. 运行回归测试（应通过）
5. 清理调试日志

### Phase 5：复盘记录

建议写入 `docs/debug/[bug-id].md`：

- 根因
- 修复点
- 测试证据
- 预防措施

---

## Full 流程切换条件

命中任意一条切换到完整调试流程：

- P0/P1 高严重问题
- 跨 3+ 模块
- 涉及迁移/兼容策略
- Lite 流程无法收敛

Full 参考入口：

- `AI开发-PLan-Program-Debug-Claude和Codex协作/06-AI调试阶段-复杂问题分析模板.md`

## 完成门禁

标记完成前必须满足：

- 复现测试通过
- 关键回归测试通过
- 无临时调试代码残留
- 向用户汇报风险与影响范围

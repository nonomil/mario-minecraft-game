# 编写技能

## 概述

**编写技能就是应用于流程文档的测试驱动开发。**

你编写测试用例（带有子代理的压力场景），看着它们失败（基线行为），编写技能（文档），看着测试通过（代理遵守），并重构（堵住漏洞）。

**核心原则：** 如果你没有在没有技能的情况下看着代理失败，你就不知道技能是否教授了正确的东西。

**必需子技能：** superpowers:test-driven-development

## 技能的 TDD 映射

| TDD 概念 | 技能创建 |
|-------------|----------------|
| **测试用例** | 带有子代理的压力场景 |
| **生产代码** | 技能文档 (SKILL.md) |
| **测试失败 (红)** | 代理在没有技能的情况下违反规则（基线） |
| **测试通过 (绿)** | 代理在有技能的情况下遵守 |
| **重构** | 在保持合规性的同时堵住漏洞 |

## 何时创建技能

**创建当：**
- 技术对你来说并不直观
- 你会在项目中再次引用这个
- 模式广泛适用（非项目特定）

**不要创建：**
- 一次性解决方案
- 项目特定的约定（放在 CLAUDE.md 中）

## SKILL.md 结构 (必需)

**YAML Frontmatter:**
- `name`: 仅字母、数字、连字符。
- `description`: 以 "Use when..." 开始，第三人称，描述触发条件。**不要总结工作流程。**

**内容：**
- `# Skill Name`
- `## Overview`: 1-2 句话核心原则。
- `## Implementation`: 代码或步骤。
- `## Common Mistakes`

## 常见错误

- **描述总结了工作流程：** 导致代理跳过阅读完整内容。
- **过度复杂的流程图：** 难以维护。
- **没有示例：** 难以理解应用。

## 危险信号

**切勿：**
- 在没有看到基线失败的情况下编写技能。
- 批量创建多个技能而不测试每个。
- 跳过 YAML frontmatter 的格式要求。

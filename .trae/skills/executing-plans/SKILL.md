# 执行计划

## 概述

执行计划技能是将静态文档转换为运行代码的过程。它强调纪律性、原子提交和频繁的状态同步。

**核心原则：** 计划即事实。

**开始时声明：** “我正在开始执行存储在 `docs/plans/<filename>.md` 中的实施计划。”

## 准备工作

1. **定位计划：** 确保你已经阅读并理解了完整的计划文件。
2. **验证环境：** 确保你在正确的 worktree 和分支上。
3. **确认基线：** 运行现有测试以确保起点是干净的。

## 执行循环

对于计划中的每个任务：

### 1. 任务声明
宣布你正在开始的任务编号和名称。

### 2. TDD 周期
严格遵循计划中定义的步骤：
- 编写失败的测试。
- 运行并确认失败。
- 编写最小代码。
- 运行并确认通过。

### 3. 原子提交
任务完成后立即提交。提交消息应引用任务编号：
`feat(auth): task 1 - implement password hashing`

### 4. 批次审查（可选）
如果是独立工作（非子代理驱动），建议每 3 个任务请求一次 superpowers:requesting-code-review。

## 状态报告

在每个任务完成或遇到阻塞时，提供简短的状态报告：

```markdown
**进度报告：**
- [x] 任务 1: 基础设置
- [/] 任务 2: 验证逻辑 (进行中)
- [ ] 任务 3: 数据持久化
```

## 遇到偏差时

如果发现计划有缺陷或需要调整：
1. **停止执行。**
2. 更新计划文档。
3. 向用户解释更改。
4. 获得批准后继续。

## 危险信号

**切勿：**
- 凭记忆执行计划（始终参照文档）。
- 跳过计划中的任何步骤（尤其是测试步骤）。
- 批量提交多个任务。
- 在未更新计划的情况下偏离计划。

## 集成

**由以下技能调用：**
- **writing-plans** - 作为并行会话执行的首选技能。
- 用户直接要求“开始执行计划”。

**使用子技能：**
- **test-driven-development** - 核心工作流程。
- **requesting-code-review** - 验证。
- **verification-before-completion** - 在结束所有任务前。
